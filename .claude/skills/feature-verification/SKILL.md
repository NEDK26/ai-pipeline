# Feature Verification：功能验证与发布门禁

当前功能：$feature_id
验证范围：$scope

## 目标

对 `$feature_id` 生成一个可追溯、可复现、可审查的验证结论。

验证不是“跑几个命令后说通过”。

必须建立以下链路：

验收用例
→ 自动化测试或人工验证步骤
→ 测试执行结果
→ 截图、日志、Trace 等证据
→ 缺陷与风险
→ 发布质量结论

---

## 严格边界

本 Skill 可以：

* 阅读功能规格、验收用例、接口契约、现有测试和代码；
* 创建或更新测试代码；
* 启动本地测试环境；
* 运行现有测试命令；
* 执行浏览器端到端测试；
* 生成测试报告、截图、Trace 和缺陷记录；
* 更新验证相关文档。

本 Skill 不可以：

* 修改业务实现代码以“修复”测试失败；
* 删除、跳过或弱化失败测试以获得通过；
* 未经人工确认更新视觉回归基线；
* 自动提交、推送、发布或部署；
* 把未执行的测试写成已通过；
* 将“无报错”当作“功能验收通过”。

若发现产品代码缺陷：

1. 记录复现条件、失败现象、日志和影响范围；
2. 在 `defects.md` 中创建缺陷记录；
3. 标记为 `Blocked` 或 `Ready with Risks`；
4. 将修复任务回流到 `feature-backend`、`feature-frontend` 或 Bug 修复流程。

## 第一步：读取规格包与现有测试

优先读取：

1. `.claude/CLAUDE.md`；
2. `docs/features/$feature_id/brief.md`；
3. `docs/features/$feature_id/spec.md`；
4. `docs/features/$feature_id/prototype.md`；
5. `docs/features/$feature_id/acceptance-tests.md`；
6. `docs/features/$feature_id/architecture-impact.md`；
7. `docs/features/$feature_id/handoff.md`；
8. 当前功能相关的后端、前端、接口与测试代码；
9. 项目已有测试配置、CI 配置、Playwright 配置和测试命令。

不要猜测项目测试命令。

必须从以下来源确认：

* `.claude/CLAUDE.md`；
* `package.json`；
* `pom.xml`、Gradle 配置或 Makefile；
* CI 配置；
* 已有测试目录和脚本。

---

## 第二步：生成验收可追溯矩阵

创建或更新：

`docs/features/$feature_id/test-traceability.md`

每条验收用例必须映射到至少一种验证方式：

| 验收用例 | 风险等级 | 验证层 | 自动化位置 / 手工步骤 | 执行状态 | 证据路径 |
| ---- | ---- | --- | ------------ | ---- | ---- |

验证层只能使用以下分类：

* Backend Unit
* Backend Integration
* API Contract
* Frontend Unit
* E2E
* Visual Review
* Manual Verification

规则：

* 高风险业务规则必须至少有 Backend Unit 或 Backend Integration 覆盖；
* 主用户路径必须至少有 E2E 覆盖；
* 高风险 UI 操作必须至少有 E2E 和截图证据；
* 仅用截图不能证明业务逻辑正确；
* 仅用接口成功不能证明用户流程可完成；
* 无法自动化的用例必须写清人工验证步骤和原因。

如果某条关键验收用例没有覆盖，标记为 `Coverage Gap`，不得宣布 Ready。

---

## 第三步：执行分层验证

按项目实际技术栈执行以下测试层。

### A. 后端测试

验证：

* 业务规则；
* 状态流转；
* 权限；
* 参数校验；
* 幂等与重复提交；
* 并发或条件更新；
* 事务边界；
* 错误码；
* 审计记录。

优先运行项目已有的单元测试和集成测试。

若发现关键规则无测试覆盖，可以新增测试代码，但不得修改业务实现使其通过。

### B. 前端测试

验证：

* 表单校验；
* 按钮禁用和提交中状态；
* 权限不足状态；
* 空数据、加载、错误状态；
* 接口错误提示；
* 高风险操作确认；
* 关键状态展示。

优先复用项目现有测试框架与组件测试模式。

### C. API 契约与联调验证

验证：

* 请求字段；
* 返回字段；
* 分页、筛选、排序；
* 业务错误码；
* 权限限制；
* 状态限制；
* 字段口径；
* 前后端对异常响应的处理一致性。

### D. 浏览器端到端验证

若项目已安装 `webapp-testing`，使用它协助：

* 启动本地前后端服务；
* 登录或准备测试数据；
* 执行关键用户路径；
* 捕获浏览器 Console 错误；
* 捕获失败网络请求；
* 对关键节点截图；
* 在失败时保留可复现证据。

若项目已有 Playwright E2E 测试，优先扩展现有测试，而不是只做一次性手工浏览器操作。

关键路径至少覆盖：

* 主流程；
* 一条权限或状态拒绝流程；
* 一条异常或失败恢复流程；
* 一条高风险操作确认流程。

### E. 视觉验证

默认只对关键页面做截图证据，不默认开启全量像素级视觉回归。

只有满足以下条件时才建立视觉快照基线：

* 页面内容可稳定复现；
* 时间、随机数据、动态广告、实时指标已固定或隐藏；
* 运行环境可固定；
* 基线截图经过人工审查。

截图差异必须人工判断原因。

禁止因为“截图不同”就自动更新基线并判定通过。

---

## 第四步：失败分类与回流

每个失败项必须归类为以下之一：

* Product Gap：需求、规则或验收标准不明确；
* Backend Defect：后端业务、接口、权限、状态或数据问题；
* Frontend Defect：页面、交互、状态展示或错误处理问题；
* Contract Mismatch：前后端接口或字段口径不一致；
* Test Defect：测试脚本、测试数据或环境问题；
* Environment Issue：依赖服务、账号、网络或环境不可用；
* Visual Review Finding：视觉层级、溢出、一致性或可用性问题。

不要在测试结论中写“有问题待优化”。

必须写明：

* 如何复现；
* 当前结果；
* 期望结果；
* 影响范围；
* 严重级别；
* 建议回流的处理阶段；
* 已保存的证据路径。

---

## 第五步：生成验证结论

创建或更新：

`docs/features/$feature_id/verification-report.md`

必须包含：

# $feature_id：验证报告

## 1. 验证范围

* 本次版本；
* 测试环境；
* 前端与后端版本；
* 数据准备方式；
* 执行时间。

## 2. 测试结果摘要

| 测试层 | 通过 | 失败 | 跳过 | 未执行 | 结论 |
| --- | -: | -: | -: | --: | -- |

## 3. 验收覆盖情况

* 已覆盖的验收用例；
* 未覆盖的验收用例；
* 仅人工验证的用例；
* 覆盖缺口原因。

## 4. 关键证据

* 测试报告路径；
* 失败截图路径；
* 关键成功路径截图；
* Trace 路径；
* Console 错误；
* 网络失败请求；
* 接口测试结果。

## 5. 缺陷与风险

* 按严重级别列出；
* 明确是否影响发布。

## 6. 发布质量结论

只能使用以下三种结论：

* Ready：无阻塞缺陷；关键验收用例已覆盖并通过；关键证据齐全。
* Ready with Risks：无阻塞缺陷，但存在已明确接受的非阻塞风险或覆盖缺口。
* Blocked：存在阻塞缺陷、关键用例失败、关键用例无覆盖，或环境导致无法完成验证。

---

## 发布门禁

只有同时满足以下条件，才能标记为 Ready：

* [ ] 所有 P0 / P1 验收用例已执行并通过；
* [ ] 所有高风险状态流转已验证；
* [ ] 权限拒绝场景已验证；
* [ ] 主用户路径已完成 E2E 验证；
* [ ] 后端关键业务规则有自动化覆盖；
* [ ] 前后端接口契约无未解决不一致；
* [ ] 无阻塞 Console 错误或失败网络请求；
* [ ] 关键截图或 Trace 证据已保存；
* [ ] 没有未说明的跳过测试；
* [ ] 所有未解决风险已写入验证报告。

---
