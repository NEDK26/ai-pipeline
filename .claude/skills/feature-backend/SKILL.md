# Feature Backend：后端功能交付

当前输入：$ARGUMENTS

## 目标

基于已经确认的 Feature 规格包，实现可测试、可联调、可审查的后端功能。

后端交付必须覆盖：

* 领域对象与业务规则；
* 状态机与合法状态流转；
* API 契约；
* 数据模型与数据访问；
* 权限控制；
* 事务、幂等与并发安全；
* 审计与错误处理；
* 单元测试、集成测试或接口测试；
* 前端联调说明。

---

## 前置条件

开始前必须读取：

1. `.claude/CLAUDE.md`；
2. `docs/features/<feature-id>/spec.md`；
3. `docs/features/<feature-id>/acceptance-tests.md`；
4. `docs/features/<feature-id>/architecture-impact.md`；
5. `docs/features/<feature-id>/handoff.md`；
6. 当前 Feature 相关的已有代码、接口、数据库表、枚举和测试；
7. 项目已有构建、测试、迁移和代码规范配置。

若 `handoff.md` 的结论不是 `Ready` 或 `Ready with Risks`：

* 不开始写业务代码；
* 不自行补全关键需求；
* 列出阻塞项；
* 建议回流到 `feature-discovery` 或 `requirements-reviewer`。

---

## 严格边界

本 Skill 可以：

* 新增或修改当前 Feature 相关后端代码；
* 新增或更新测试；
* 新增必要的数据迁移；
* 更新 API 契约和后端联调说明；
* 执行项目已有构建、测试和静态检查命令。

本 Skill 不可以：

* 擅自改变产品范围；
* 自行修改关键业务规则；
* 绕开已有领域模型或状态机；
* 为了通过测试删除断言、跳过测试或弱化校验；
* 修改无关模块；
* 自动提交、推送、发布或执行生产数据库操作；
* 将前端校验视为后端校验的替代品。

## 第一步：建立实现映射

先创建或更新：

`docs/features/<feature-id>/backend-handoff.md`

内容至少包含：

| 规格项         | 后端落点                           | 测试落点                     | 状态             |
| ----------- | ------------------------------ | ------------------------ | -------------- |
| 用户故事 / 业务规则 | Service / Domain / Controller  | Unit / Integration / API | Pending / Done |
| 权限规则        | 权限校验位置                         | 权限测试                     | Pending / Done |
| 状态流转        | Enum / State Machine / Service | 状态机测试                    | Pending / Done |
| 数据字段        | Entity / DTO / Migration       | 数据校验测试                   | Pending / Done |
| 审计要求        | Audit Log / Operation Log      | 审计测试                     | Pending / Done |

每个重要业务规则必须有明确后端落点。

禁止出现：

```text
“前端已限制，因此后端无需校验”
```

## 第二步：先定义后端设计

在修改代码前，确认以下内容。

### 领域与数据

* 需要新增或调整哪些 Entity、DTO、VO、Enum；
* 是否需要数据库字段、索引、唯一约束或迁移；
* 字段是否存在空值、默认值、历史数据兼容问题；
* 是否需要审计字段；
* 是否需要逻辑删除、版本号或乐观锁字段。

### 状态机

涉及发布、回滚、删除、审批、退款、库存、名额、任务执行等状态变化时：

* 定义状态枚举；
* 定义允许转移；
* 定义禁止转移；
* 定义转移前置条件；
* 定义失败后的状态；
* 定义回退或补偿方式；
* 禁止 Controller 直接更新状态字段。

### API 契约

明确：

* 请求路径与方法；
* 请求 DTO；
* 返回 VO；
* 参数校验；
* 权限要求；
* 成功响应；
* 参数错误、权限不足、状态不允许、资源不存在等错误响应；
* 分页、筛选、排序和时间字段口径；
* 幂等要求。

---

## 第三步：按 TDD 实现

优先使用项目已有测试框架与测试风格。

实现顺序：

1. 为关键业务规则写失败测试；
2. 实现最小可通过业务逻辑；
3. 运行测试；
4. 重构重复代码；
5. 继续下一条规格规则。

至少优先覆盖：

* 正常主流程；
* 参数校验；
* 权限不足；
* 状态不允许；
* 重复提交；
* 并发或条件更新；
* 外部依赖失败；
* 审计记录；
* 回滚或恢复。

---

## 第四步：实现要求

### Controller

* 只负责请求接收、DTO 校验、权限入口和统一响应；
* 不写复杂业务规则；
* 不直接调用 Mapper；
* 不返回 Entity；
* 不暴露内部异常细节。

### Service

* 承担业务规则、状态校验、事务边界和审计逻辑；
* 涉及多表修改、状态切换、金额、库存、发布、回滚时，评估并正确使用事务；
* 失败时保证不会留下半完成数据；
* 不允许绕过状态机直接更新关键状态。

### Mapper / Repository

* 仅负责数据访问；
* 更新和删除必须有明确条件；
* 禁止无条件批量更新、删除；
* 分页查询必须有稳定排序；
* 避免 `select *`；
* 检查可能的 N+1 查询和索引缺失。

### 金额、时间与枚举

* 金额使用 `BigDecimal`，比较使用 `compareTo`；
* 时间优先使用项目既有的 `LocalDateTime` 规范；
* 状态、类型、策略使用 Enum；
* 禁止新增魔法数字和散落字符串状态。

### 权限与审计

高风险操作，例如：

* 发布；
* 停止发布；
* 回滚；
* 删除；
* 权限调整；
* 强制更新；
* 金额变更；

必须同时满足：

* 后端权限校验；
* 状态前置校验；
* 操作影响范围明确；
* 审计记录完整；
* 必要时二次确认参数或幂等键；
* 可追踪操作人、时间、对象和结果。

---

## 第五步：测试与联调

执行项目真实测试命令，不要猜命令。

至少验证：

### 单元测试

* Service 关键规则；
* 状态转移；
* 权限拒绝；
* 参数校验；
* 重复调用或幂等；
* 审计记录。

### 集成 / 接口测试

* Controller 到数据库的主路径；
* 请求和返回字段；
* 业务错误码；
* 权限与状态限制；
* 数据库写入结果；
* 异常回滚。

### 前端联调说明

在 `backend-handoff.md` 补充：

* 已实现接口；
* 请求参数；
* 返回字段；
* 错误码；
* 空值与默认值语义；
* 分页与排序规则；
* 接口是否已可供前端联调；
* 尚未完成或存在风险的接口。

---

## 第六步：完成标准

满足以下条件才可标记为后端实现完成：

* [ ] 每条关键业务规则都有后端实现落点；
* [ ] 状态流转不可被直接绕过；
* [ ] 高风险操作有后端权限校验；
* [ ] 关键写操作具备合理事务边界；
* [ ] 重复请求与并发风险已评估；
* [ ] 数据更新、删除均有明确条件；
* [ ] API 契约与规格一致；
* [ ] 关键业务规则已有自动化测试；
* [ ] 测试命令已真实执行；
* [ ] 前端联调说明已更新；
* [ ] 没有为了通过测试而削弱业务校验。

若存在未解决风险，必须明确写入 `backend-handoff.md`，不能把状态标记为无风险完成。
